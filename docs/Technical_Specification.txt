Smart Driver Scoring System: Technical Specification

This document details the low-level technical implementation of the Smart Driver Scoring System, covering hardware communication, data processing pipelines, and machine learning architectures.

---

## 1. Data Acquisition (OBD-II & BLE)

The system communicates with the vehicle's ECU via an ELM327-compatible Bluetooth Low Energy (BLE) adapter using the GATT protocol.

### BLE GATT Configuration
- Service UUID: Standard Serial Port Profile (SPP) over BLE variants.
- TX Characteristic (Write): 0000fff2-0000-1000-8000-00805f9b34fb
- RX Characteristic (Notify): 0000fff1-0000-1000-8000-00805f9b34fb

### ELM327 Initialization Sequence
1.  AT Z: Hard reset.
2.  AT E0: Echo off (reduces bandwidth).
3.  AT SP 0: Automatic protocol detection (ISO 15765-4, etc.).
4.  AT H0: Headers off (cleaner PID responses).

### OBD-II PID Mapping & Formulas
- Engine RPM (PID 01 0C): 2 bytes, Formula: ((A * 256) + B) / 4.0
- Vehicle Speed (PID 01 0D): 1 byte, Formula: A
- Throttle Position (PID 01 11): 1 byte, Formula: (A * 100.0) / 255.0
- Engine Load (PID 01 04): 1 byte, Formula: (A * 100.0) / 255.0

## 2. Feature Engineering Pipeline

The raw sensor data is processed through a FeatureExtractor unit that maintains a 10-sample rolling buffer (deque).

### Feature Categories (25 Total Features)
1.  Instantaneous: Speed, RPM, Throttle, Engine Load.
2.  Derivative (Delta):
    *   speed_change: Abs(Vt - Vt-1)
    *   throttle_jerk: Abs(Tt - Tt-1)
    *   rpm_change: Abs(Rt - Rt-1)
3.  Relational: rpm_speed_ratio (R/V, clipped at V=1).
4.  Rolling Statistics (5 & 10 samples):
    *   Mean: Arithmetic average of the sub-window.
    *   Standard Deviation: Calculation of dispersion for stability detection.

## 3. Machine Learning Architecture

The "Balanced Model" is a gradient-boosted decision tree ensemble optimized for embedded hardware efficiency.

### Model Specification
- Algorithm: HistGradientBoostingClassifier (Scikit-Learn).
- Learning Rate: 0.1
- Max Iterations: 200
- Class Balancing: class_weight='balanced' (assigns higher cost to harsh driving events to handle data imbalance).
- Inference Time: < 50ms per sample on Raspberry Pi 4.

### Training Labeling Logic (Boolean Mask)
A sample is labeled Aggressive (1) if any of these conditions are met:
1.  Aggressive Take-off: (RPM > 2700) AND (Throttle > 35%)
2.  High Engine Strain: (Engine Load > 92%) AND (Throttle > 35%)
3.  Hard Braking/Accel: Speed Change > 12 km/h per sample

## 4. System Operation & UI

The application utilizes a multi-threaded architecture to ensure UI responsiveness.

### Threading Model
- Main Thread: Handles the Tkinter Event Loop (100ms refresh rate via .after()).
- Data Thread: Background worker for BLE communication and ML inference.
- Inter-process Communication (IPC): Python queue.Queue for thread-safe data transfer.

### UI Specifications
- Resolution: 480x320 pixels (optimized for 3.5" RPi screens).
- Communication Rate: Queries OBD adapter every 500ms.
- State Management: Only updates UI components if the underlying value changes (Atomic updates) to prevent flickering and save CPU cycles.

---

## 5. Data Persistence
Every trip is recorded as a CSV in data/trips/ with the following schema:
- timestamp, speed, rpm, throttle, engine_load, score, prediction.

IMPORTANT: The system implements a Mock Mode using src/acquisition/sim.py which replays historical CSV data for testing without car access.
